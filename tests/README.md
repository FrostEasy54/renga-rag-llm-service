# tests/ — трёхрукий эксперимент к главе 2.2 диплома

Три конфигурации:

| Имя | `mode` (API) | Что включает |
|---|---|---|
| **BARE** | `bare` | Минимальный промпт. Системный промпт + данные о здании + JSON-схема. Никакого знания о строительстве. |
| **STRUCTURED** | `structured` | Структурный промпт: 4 этапа МДС 12-43.2008, поэтажная развёртка, 10 правил последовательности. **Без RAG.** |
| **RAG** | `rag` | Структурный промпт + RAG-контекст из базы знаний. Это продакшен-режим. |

Три парных сравнения:
- **STRUCTURED vs BARE** — вклад структурного промпта (что даёт инженерная проработка инструкции).
- **RAG vs STRUCTURED** — вклад RAG поверх структурного промпта (изолированный эффект ретривала).
- **RAG vs BARE** — совокупный вклад системы.

## Установка

```powershell
cd tests
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

## Запуск

В одном терминале — backend:

```powershell
cd rag-llm-service
just run
```

В другом — эксперимент:

```powershell
cd rag-llm-service/tests

# Smoke-тест: 1 модель × 3 конфига × 1 прогон ≈ 3-4 минуты
python run_experiment.py --smoke

# Полный прогон по умолчанию: 15 моделей × 3 конфига × 3 прогона = 135 запросов
# Время: ~2.5-3 часа на RTX 3050 8 GB VRAM
python run_experiment.py

# Подвыборка моделей через CLI — для быстрой итерации:
python run_experiment.py --models m01,m06,m11
```

Тестовый набор — 15 моделей из manifest.json, по 5 в каждой группе сложности:

| Группа | Модели | Параметры |
|---|---|---|
| Малая | m01–m05 | 1–2 этажа, 80–250 м² |
| Средняя | m06–m10 | 3–9 этажей, 1800–5800 м² |
| Высокая | m11–m15 | 10–17 этажей, 8500–25000 м² |

При N=15 минимальное p-значение одностороннего критерия Уилкоксона очень мало (порядка 10⁻⁵) — статистическая мощность для отвержения H₀ при α=0.05 практически не лимитирована.

## Что появится после прогона

```
tests/
├── responses/
│   ├── t001.json … t135.json     ← сырые ответы LLM
└── results/
    ├── tests.csv                  ← 135 строк: каждый запуск
    ├── per_building.csv           ← 15 строк: агрегат по модели × конфигу
    └── summary.csv                ← 4 строки (по метрике): омнибус + парные сравнения
```

`tests.csv` пишется **инкрементально** после каждого запроса — прогресс не теряется при сбое.

## Главный результат — `summary.csv`

4 строки (метрики C, S, D, Q), каждая с:

- `mean_BARE`, `mean_STRUCTURED`, `mean_RAG`, `std_*` — средние и SD по 15 моделям;
- `delta_struct_vs_bare`, `delta_rag_vs_struct`, `delta_rag_vs_bare` — разности средних;
- `friedman_chi2`, `friedman_p` — омнибусный тест Фридмана (есть ли различия вообще);
- `wilcoxon_p_*` и `t_test_p_*` — парные сравнения, односторонняя альтернатива;
- `cohen_dz_*` — величины эффекта для каждого парного сравнения;
- `friedman_significant_005` — флаг значимости омнибуса.

**Прогнозируемый паттерн** (по подразделу «Границы сравнения» главы 2.2):
- На метриках C и S: больший прирост на `delta_struct_vs_bare`, меньший — на `delta_rag_vs_struct`. Структурный промпт делает основную работу для C/S, RAG доделывает только нюансы.
- На метрике D: больший прирост на `delta_rag_vs_struct`. Конкретные нормативные значения T_норм приходят только через RAG.

Если этот паттерн подтвердится — будет хороший аргумент для дискуссии: структурный промпт и RAG дают **разный тип знания** (структурное vs количественное), и оба нужны.

## Если что-то пошло не так

| Проблема | Что делать |
|---|---|
| Скрипт упал на N-м тесте | Первые N-1 строк в `tests.csv`. Удалить ровно сбойные ответы из `responses/` и запустить снова — или перепрогнать только нужные `--models`. |
| `422 Unprocessable Entity` | LLM вернула невалидный JSON. Смотреть `responses/<id>.json` руками. Часто помогает повторить запрос (стохастика). |
| Долго (>2 мин на запрос) | Ollama выгрузила модель из VRAM. Повторный запрос вернёт её. На backend-е это обработано через автоповтор. |
| Хочу полностью пересчитать метрики из существующих ответов | На текущий момент не реализовано отдельной командой. Если потребуется — добавлю `reaggregate.py`. |
